## 一句话定义

`lakejobai-job-radar` 是一个面向个人求职的 BOSS 直聘自动化助手：它把岗位搜索、福利筛选、批量投递、HR 会话记录、AI 回复和本地数据沉淀做成一个 Web 控制台，同时提供 `lakejob` CLI 方便 AI Agent 调用。

## 它可以用来干什么

- 自动搜索 BOSS 直聘岗位：按关键词、城市、薪资、经验、学历等条件拉取岗位。
- 做岗位初筛：保留薪资、公司、城市、福利标签等信息，适合快速比较岗位质量。
- 批量投递岗位：把搜索到的待投递岗位按队列执行投递，并统计当天投递数量。
- 维护本地求职记录：用 SQLite 记录已发现岗位、投递状态、会话、收藏池和统计数据。
- 辅助回复 HR：配置 AI Key 后，可以生成更职业化的招呼语和聊天回复。
- 作为 Agent 工具：CLI 输出 JSON，适合被 Codex、Claude Code 或其他自动化 Agent 调用，比如 `lakejob status`、`lakejob search`、`lakejob apply-batch`。

## 适合的使用场景

- 找实习、校招、社招时，需要每天重复搜索同类岗位。
- 想把求职过程从“手动刷岗位”变成“搜索、筛选、投递、复盘”的半自动流程。
- 想让 AI Agent 帮忙做求职数据整理、岗位比较、投递进度统计。
- 想在本地保留求职漏斗，而不是完全依赖招聘平台的历史记录。

不适合的场景：

- 批量营销、商业采集、绕过平台风控。
- 多账号自动化注册或高频投递。
- 完全无人值守地代替真人沟通。

## 关键能力拆解

### Web 控制台

网页控制台是日常使用入口，适合做：

- 查看浏览器和登录状态。
- 配置 AI 模型。
- 搜索岗位。
- 查看岗位列表和投递进度。
- 查看 HR 会话。
- 管理收藏岗位和设置。

### 浏览器自动化

项目使用 Playwright + Firefox 操作 BOSS 直聘网页。它不是官方 API，所以稳定性取决于：

- BOSS 页面结构是否变化。
- 当前登录态是否有效。
- 平台是否触发风控。
- 浏览器 Profile 是否保持正常。

### 本地状态库

项目使用 SQLite 记录求职过程数据。这个价值很大，因为它能把“刷岗位”变成可复盘的数据：

- 今天投了多少。
- 哪些岗位还待投。
- 哪些岗位已经沟通过。
- 哪些岗位值得收藏。
- 后续可以做岗位质量分析和投递转化漏斗。

### AI 回复

AI 不是核心依赖，但可以增强沟通效率：

- 根据简历摘要生成招呼语。
- 根据 HR 信息生成职业化回复。
- 分析岗位 JD 和个人背景匹配度。

使用时要保留人工确认，尤其是发送个人信息、微信、电话、简历之前。

## 合规和风险边界

这个项目只能作为个人求职辅助工具使用。使用时需要守住几个边界：

- 控制每日投递上限，避免刷屏式投递。
- 不做商业采集，不导出平台数据做外部售卖。
- 不绕过平台登录、验证码、风控。
- 自动回复前尽量人工确认，避免向 HR 发送错误或过度承诺的信息。
- 个人隐私信息只在必要时发送，尤其是手机号、微信、简历和学校信息。

## 与我的知识库/Agent 工作流的关系

这个项目可以作为“浏览器自动化 + 本地数据沉淀 + AI Agent 调用”的一个具体样板。

它证明了一件事：Agent 不一定只是在聊天窗口里回答问题，也可以通过 CLI、Web 服务和浏览器自动化接入真实流程。对我的知识库来说，它可以沉淀三类经验：

- 求职流程自动化：把岗位搜索、筛选、投递、复盘结构化。
- Agent 工具接口设计：CLI 输出 JSON，方便其他 Agent 调用。
- 自动化边界治理：真实账号、真实平台、真实个人信息都需要权限、日志和人工确认。
