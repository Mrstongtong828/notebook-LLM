## 一、核心概念篇

### 1.1 Harness Engineering — 套件工程

> **核心矛盾**：上下文窗口有限 vs 任务复杂度无限。解法不是更大的窗口，而是更聪明的上下文管理。

Harness Engineering 的本质：不是让 AI 一次做完所有事，而是设计一套"套件(harness)"——**决定存储什么、检索什么、展示给模型什么**。主智能体就是这个套件的编排者。

**思维转变**：你不是在"写 prompt"，你是在设计一个系统——决定信息如何流转、如何隔离、如何持久化。

### 1.2 三条铁律

| 铁律 | 含义 | 违反后果 |
|------|------|----------|
| **文件即记忆** | 子智能体的所有产出必须持久化到文件，不依赖内存 | 上下文丢失后无法恢复 |
| **隔离即常态** | 每个子智能体只看到主智能体给它的信息，不假设它知道外部状态 | 隐式依赖导致幻觉 |
| **记录即保险** | 主智能体和子智能体都写日志，确保一切可追溯 | 出问题后无法复盘 |

> 💡 **面试提炼**：这三条铁律回答的是"如何解决 LLM 上下文窗口限制"这个核心问题。你能讲出"File-based Memory"和"信息隔离"，面试官会认为你踩过生产的坑。

### 1.3 三角色模型

```
┌─────────────┐
│   主智能体    │  编排者：拆任务、管状态、中转信息
│  (Orchestrator) │
└──────┬──────┘
       │
   ┌───┴───────────────┐
   │                   │
   ▼                   ▼
┌──────────┐     ┌──────────┐
│ 开发智能体 │     │ 测试智能体 │
│  (Dev)   │     │ (Tester)  │
└──────────┘     └──────────┘
```

| 角色 | 核心职责 | 工具权限 | 是否可修改代码 |
|------|---------|----------|:------------:|
| **主智能体 (Orchestrator)** | 拆任务、管计划、中转信息、写日志 | 全部（调度用） | ❌ |
| **开发智能体 (Developer)** | 读需求→写代码→自测→输出 | Read + Edit + Write + Bash | ✅ |
| **测试智能体 (Tester)** | 读需求→审查代码→输出结论 | Read + Bash + Grep **（禁 Edit/Write）** | ❌ |

> 🔑 **关键设计原则**：
> - 测试智能体**必须禁用 Edit/Write** 权限，防止"越权"
> - 开发智能体完成一个子任务后**不保留上下文**，下次新任务重新创建
> - 只有修正循环时才 **resume** 之前的智能体（保留历史上下文）

## 二、Orchestrator 模式详解

> 实战案例：一个 HTML 幻灯片多智能体开发系统，主Agent 协调 Planner、Dev、3 个 Tester（Layout/Beauty/Animation）

### 2.1 核心原则：主Agent 只调度不干活

```
主Agent 的禁忌清单：
❌ 不直接编辑任何代码文件
❌ 不读素材文件内容（只传路径给子Agent）
❌ 不读测试报告内容（只用 Grep 提取 PASS/FAIL 判定）
❌ 不对后台通知做详细回应（回复"已确认"三字即可）
```

**为什么**：主Agent 的上下文是系统中最宝贵的资源。每多读一份文件，就少一分注意力给调度决策。

### 2.2 Agent ID 管理（核心操作）

每次创建子Agent 后，**必须立即获取其 Agent ID**，用于后续 resume：

```bash
# 查找最新的 Agent meta 文件
ls -lt ~/.claude/projects/*/*/subagents/agent-*.meta.json | head -3

# 或
find ~/.claude/projects -name "agent-*.meta.json" -type f -printf '%T@ %p\n' | sort -rn | head -5
```

**ID 使用规则**：
- Resume 用**裸 ID**（`a95e84cd`），不带 `agent-` 前缀和 `.meta.json` 后缀
- Resume **必须指定 subagent_type**，与创建时一致
- 每轮开发结束，DEV_ID 失效；同页修正循环中**复用**同一 DEV_ID

### 2.3 开发-测试-修正循环（核心流程）

```
每个子任务的标准操作序列：

1️⃣ 创建开发子Agent → 等完成 → 提取 DEV_ID
2️⃣ 创建测试子Agent → 等完成 → 提取 TEST_ID
3️⃣ 判定：
   ✅ PASS → 更新计划 → 进入下一任务
   ❌ FAIL → 进入修正循环（最多 3-5 轮）

修正循环：
┌─ resume 开发子Agent（用 DEV_ID）→ 修改代码
├─ resume 测试子Agent（用 TEST_ID）→ 重新审查
└─ 回到判定（PASS 则退出，FAIL 则继续）
```

**并发设计**（来自实战案例）：
- 开发阶段：每批只启动 **1 个** 开发Agent
- 测试阶段：**3 个** 测试Agent 并行运行（布局/美观/动画各一个）
- 并发上限始终 ≤ 3

### 2.4 上下文保护策略

| 策略 | 做法 | 收益 |
|------|------|------|
| **只传路径不传内容** | 素材文件只传路径，子Agent 自己读 | 主Agent 上下文不被大文件撑爆 |
| **只提取判定结果** | 测试报告用 `Grep "^### 判定"` 取 PASS/FAIL | 不读完整报告 |
| **批量处理** | N 页一批，开发/测试同批量 | 减少调度轮次 |
| **后台通知简短确认** | 回复"已确认"三字 | 不打断主Agent 的思维流 |

> 💡 **面试价值**：能讲出"上下文保护"这个话题，说明你理解生产环境 Agent 的核心挑战。

## 三、子Agent 模板设计

### 3.1 模板通用结构

一个子Agent 模板文件应包含：

```yaml
---
name: agent-name           # 智能体名称
description: |             # 触发场景描述
  什么情况下触发这个Agent...
tools: Read, Edit, Write, Bash, Glob, Grep  # 工具权限
model: inherit / haiku / sonnet              # 模型选择
permissionMode: acceptEdits                  # 权限模式
memory: project                              # 记忆模式
skills:                                      # 预加载 skill
  - skill-name
---

## 角色定义
{一句话说明你是谁、干什么}

## 工作流程
{按步骤列出执行过程}

## 必读文件（按顺序）
{列出每次任务开始时必须读取的文件}

## 输出格式
{强制要求的输出模板，不得违反}
```

### 3.2 工具权限设计模式

| 角色 | 允许的工具 | 禁止的工具 | 理由 |
|------|-----------|-----------|------|
| **开发Agent** | Read + Edit + Write + Bash + Glob + Grep | — | 需要完整开发能力 |
| **测试Agent** | Read + Bash + Glob + Grep | **Edit, Write** | 只读，防止越权修改 |

> 🔑 **面试题**："如何防止测试Agent 偷偷改代码？" → 在模板层面禁用 Edit/Write 工具。

### 3.3 输入输出契约

**关键原则：子Agent 的返回值必须极简，只返回路径和判定，不返回内容。**

```
✅ 正确示例（测试Agent 返回）：
   测试结果：PASS
   报告路径：test-reports/page01-layout.md

❌ 错误示例：
   测试结果：PASS，经过对代码的详细审查...
```

**为什么**：子Agent 的返回值会直接进入主Agent 的上下文。主Agent 每多读一个字，调度能力就少一分。

### 3.4 模型选择策略

| 场景 | 推荐模型 | 理由 |
|------|----------|------|
| 简单、重复性任务 | haiku (cheap) | 快速、便宜、够用 |
| 代码开发 | inherit（跟随主模型） | 开发需要最强推理 |
| 测试/审查 | haiku 或 sonnet | 需要足够的判断力 |
| 关键决策 | sonnet / opus | 最强推理能力 |

**2026 升级版**（来自 Hermes Agent 实践）：
- 核心对话 → **贵模型**（如 MiniMax-M2.7 / GPT-4）
- 辅助任务（摘要、审批、标题生成）→ **便宜模型**（如 Gemini 2.5 Flash）
- 通过主副模型分工，成本降低 **60-80%**

## 四、File-based Memory 系统

### 4.1 三层文件架构

```
项目根目录/
├── doc/
│   ├── plan.md               # 开发计划（主Agent 维护）
│   ├── requirements.md       # 需求文档
│   ├── lessons-learned.md    # 经验库（跨任务积累）
│   └── design-guide.md       # 设计指南
├── src/                      # 代码目录
├── logs/                     # 日志目录
└── test-reports/             # 测试报告目录
```

### 4.2 经验库设计

经验库是**跨子智能体传递知识的唯一可靠通道**。它的价值在于让后续的Agent 不重复踩坑。

**经验总结的粒度原则**：
- ❌ **太细**："Page03 第 42 行的 left 值应该是 120 而不是 100" → 只适用于一个像素
- ❌ **太宽**："注意布局要好看" → 没有可操作性
- ✅ **恰好**："标题和正文字号差距至少 8px" → 有具体标准，适用于所有场景

**判断标准**：如果去掉具体数值和页面名，这句话还能指导决策吗？如果不能，就还没抽象到位。

### 4.3 三层日志架构

```
主智能体日志（全流程）
  │
  ├── 子智能体日志（各自独立）
  │     ├── 开发智能体日志
  │     └── 测试智能体日志
  │
  └── 经验库（跨任务积累）
```

**主智能体日志格式**：记录每个关键事件的**时间 + 子智能体ID + 结果**，便于回溯。

```
- 260424 2342 开发完成：page01 section 追加 (DEV_ID: a95e84cd)
- 260424 2344 首次测试 page01：布局PASS / 美观FAIL / 动画PASS
```

## 五、工程最佳实践

### 5.1 上下文管理策略对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **一个长对话** | 上下文完整 | 窗口爆满，注意力分散 | 简单任务 |
| **每任务新智能体** | 上下文干净 | 丢失历史，重复阅读 | 独立子任务 |
| **新智能体 + 文件传递** ⭐ | 干净且有记录 | 需要精心设计传入内容 | **复杂多步骤任务** |

### 5.2 任务拆分粒度

| 太粗 | 恰好 | 太细 |
|------|------|------|
| "做完整套动画" | "开发 Page03 - 数据图表页" | "写 Page03 第 12 行的 interpolate" |
| 一个任务 > 500 行代码 | 一个任务 100-300 行代码 | 一个任务 < 30 行代码 |
| 难以测试 | **可独立验证** | 上下文切换成本太高 |

**黄金法则**：一个子任务 = 一个可独立运行和验证的功能单元。

### 5.3 开发阶段划分

```
Phase 0: 框架搭建      → 主Agent 直接完成（或 Planner Agent）
Phase 1~N: 逐功能开发  → 每个子任务走完整开发-测试循环
Phase Last: 整体集成    → 全局一致性验证
```

### 5.4 错误处理与容错

| 常见问题 | 原因 | 解法 |
|---------|------|------|
| Agent(resume) 找不到 ID | 路径不对 | 用 `find` + `sort -rn` 找到最新的 meta 文件 |
| Resume 报 "Agent type not found" | 未指定 subagent_type | 必须指定与创建时一致的 subagent_type |
| 子Agent 回复不在 UI 显示 | 回复在 JSONL 文件中 | 读 JSONL 文件查看 |
| 修正超 5 轮仍未通过 | 任务太难或测试标准不合理 | 标记为"低质量通过"，人工介入 |
| 开发Agent 重复踩坑 | 没读经验库 | 在 prompt 中强制要求先读 lessons-learned.md |

| 2 | **Orchestrator 模式** | 主智能体提示词 | "主Agent 只调度不干活，保持自己的上下文干净，把具体工作委托给子Agent" |

### 🎯 简历亮点话术

**"设计并实现了一个多智能体协作系统"**：
> 采用 Orchestrator 模式，主Agent 负责任务拆分和状态管理，子Agent（开发/测试）各司其职。通过文件系统实现跨 Agent 记忆共享，设计了三级日志体系确保可追溯。支持开发-测试-修正自动化循环，并发上限 3 个测试Agent 并行。上下文保护策略使主Agent 上下文始终可控。

### 🎯 能回答的追问

面试官可能会追问：
- **"上下文窗口不够怎么办？"** → 不是换更大的窗口，而是用文件传递+信息隔离
- **"Agent 重复犯错怎么办？"** → 经验库 + prompt 强制要求先读 lessons-learned.md
- **"测试Agent 会不会改代码？"** → 模板层禁用 Edit/Write 工具权限
- **"多个 Agent 怎么协同？"** → 主Agent 编排 + 文件传递 + Agent ID resume 机制
